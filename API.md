# Архитектура API Eva OS

Этот документ описывает двухуровневую архитектуру API в Eva OS. Для обеспечения максимальной безопасности, производительности и изоляции модулей, API разделено на **Внутренний Контур (gRPC)** и **Внешний Шлюз (REST/WebSocket)**.

## 1. Внутренний Контур (Core gRPC)

Ядро (`eva-daemon`) не имеет сетевого стека и не слушает TCP-порты. Вместо этого оно поднимает **Unix Domain Socket (UDS)** по пути `/tmp/eva.sock`. Общение происходит строго по протоколу gRPC (Protobuf).

### Почему UDS + gRPC?
*   **Скорость:** Вызов через UDS обходится без накладных расходов сетевого стека TCP/IP (задержки измеряются микросекундами).
*   **Безопасность:** Доступ к `/tmp/eva.sock` регулируется правами файловой системы (chmod). Никто из внешней сети не может достучаться до Ядра.
*   **Строгая типизация:** Контракты жестко заданы в `shared-ipc/proto/eva.proto`.

### Доступные gRPC RPC (`service Hypervisor`):
*   `SubmitTask(SubmitRequest)`: Передает задачу планировщику (WSJF).
*   `StreamTaskStatus(TaskStatusRequest)`: Открывает Server-Side поток для стриминга логов и токенов LLM.
*   `GetQueue(QueueRequest)`: Возвращает полное состояние DAG-графов и очередей (для системного мониторинга).
*   `GetRegistries(RegistryRequest)`: Возвращает загруженные в Ядро модели и MCP инструменты.

## 2. Внешний Шлюз (Eva API Gateway)

`eva-api` — это отдельный легковесный микросервис (написан на `axum`).
Его задача — быть фасадом между Ядром и внешним миром (Web UI, Python SDK).

### Как это работает:
1. Пользователь заходит на `http://localhost:8080/`. `eva-api` отдает статические файлы Svelte (UI).
2. UI отправляет REST запрос `POST /api/v1/tasks`.
3. `eva-api` берет JSON, сериализует его в Protobuf, открывает UDS сокет `/tmp/eva.sock` и делает сверхбыстрый gRPC вызов к Ядру.
4. Ядро ставит задачу в очередь, `eva-api` возвращает HTTP 202 Accepted.

### REST Эндпоинты (Транслируются в gRPC):
*   **`POST /api/v1/tasks`** — Запуск задачи (Fast Track / Deep Track).
*   **`GET /api/v1/tasks/{id}/stream`** — WebSocket/SSE стрим для чата.
*   **`GET /api/v1/scheduler/queue`** — Получение состояния очередей для Zabbix UI.
*   **`GET /api/v1/scheduler/dag/{id}`** — Топология конкретного графа.
*   **`GET /api/v1/telemetry/models`** — Инспекция VRAM и RAM (статус загрузки весов, MoE гибриды, $S_{penalty}$).
*   **`GET /api/v1/registry/models`** — Список доступных моделей.

### Выключение:
Поскольку `eva-api` — отдельный демон, его можно безопасно остановить (`systemctl stop eva-api`). При этом Ядро (`eva-daemon`) продолжит бесперебойно работать, обслуживая фоновые графы через IPC/shmem.
