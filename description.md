### **I. Стек и базовые решения**

* **Язык:** Rust (для Ядра и Воркеров).  
* **IPC (Межпроцессное взаимодействие):** Shared Memory (shmem) \+ Ring Buffers (shared\_memory или iceoryx).  
* **Хранилище состояний (State Persistence):** sled или rocksdb.  
* **Графы и маршрутизация:** petgraph (DAG), serde\_yaml (парсер темплейтов).  
* **API/Web:** axum \+ tokio.

### **II. Движки для Воркеров (Варианты)**

Написание обертки для проброса shmem прямо в инференс.

* **candle (HuggingFace).** Чистый Rust. Тензоры и KV-кэш крутятся в памяти воркера, входные промпты и выходные токены пишутся напрямую в shmem-буфер. Идеальная совместимость.

### **III. Архитектура Ядра (Core Daemon)**

Ядро разбито на изолированные асинхронные подсистемы.  
**1\. core::api**

* Принимает задачи от пользователя или демонов (REST/gRPC). На входе либо Template\_ID, либо сырой Task\_String.  
* Возвращает потоковые ответы.

**2\. core::router (Маршрутизатор)**

* **Узел Зеро (Zero-Node):** Если Template\_ID не передан, поднимает сверхлегкую модель для классификации задачи, присваивает ей существующий ID или тег new\_dynamic\_task, после чего немедленно выгружает модель.  
* Направляет задачу в **Fast Track** (готовые YAML-темплейты) или **Deep Track** (автономная генерация).

**3\. core::scheduler (Планировщик и Графы)**

* Строит DAG (Directed Acyclic Graph) задач на основе единой схемы PipelineDefinition.  
* Реализует слой валидации: жестко отклоняет JSON от нейронок, если он не соответствует схеме графа.  
* Реализует вытесняющую многозадачность по формуле WSJF.  
* Триггерит Воркеры на загрузку весов или сброс на CPU.

**4\. core::context\_engine (Оркестратор Контекста)**

* Управляет RAG-базой (встроенной qdrant / hnsw\_rs).  
* Хранит жесткие ограничения инструментов в .md файлах. Подает в системный промпт нейронки только релевантный список MCP-серверов, сужая фокус (Attention).

**5\. core::mcp\_runtime (Исполнитель команд)**

* Отрабатывает запросы от context\_engine (curl, nix-shell, journalctl, чтение файлов).  
* Возвращает сырые данные в Ядро.

**6\. core::state**

* Логирует этапы графа, веса и метрики API в KV-базу.  
* Хранит константы аппаратного профилирования (Init Benchmark), собранные при первой установке (PCIe throughput, IOPS диска).

### **IV. Структура декларативного пайплайна (Schema)**

Единый формат для статических YAML и динамических JSON.

Rust  
pub struct PipelineDefinition {  
    pub task\_priority: u8, // От 0 до 9  
    pub nodes: HashMap\<String, ExecutionNode\>,  
}

pub struct ExecutionNode {  
    pub dependencies: Vec\<String\>, // Для топологической сортировки DAG  
    pub action\_type: ActionType, // MCP\_Call, LLM\_Inference, Sub\_Agent, Condition\_If\_Else  
    pub target\_model: Option\<String\>, // Для расчета Spenalty  
    pub system\_prompt: Option\<String\>, // Кастомная инструкция  
    pub payload: String, // Команда или текст  
}

### **V. Архитектура Воркера**

* **Инициализация:** Запускается Ядром. Получает дескриптор shmem-сегмента.  
* **Цикл ожидания:** Висит в RAM, слушает атомарный флаг.  
* **Выполнение:** Читает команду $\\rightarrow$ генерирует $\\rightarrow$ пишет токены в кольцевой буфер $\\rightarrow$ устанавливает флаг готовности.  
* **Изоляция:** Нет сетевого стека, нет прямого доступа к FS. Строгий запрос данных у Ядра через флаг REQ\_DATA.

### **VI. Математический аппарат (Формулы планировщика)**

**1\. Динамический приоритет узла (WSJF):**  
Вычисляется каждую итерацию для узлов в Ready.

$$Score \= \\frac{P\_{task} \+ W\_{aging}}{T\_{estimate} \+ S\_{penalty}}$$

* $P\_{task}$: Базовый приоритет графа (от 0 до 9).  
* $W\_{aging}$: Время нахождения узла в ожидании $\\times$ коэффициент старения.  
* $T\_{estimate}$: Стоимость выполнения (Cost Units, рассчитанные на базе констант Init Benchmark).  
* $S\_{penalty}$: Штраф Context Switch (0, если нужная модель в VRAM; $X$ Cost Units, если нужна загрузка весов).

**2\. Стоимость удаленного API ($T\_{estimate}$ для Remote LLM):**

$$T\_{remote} \= Network\\\_Cost \+ TTFT\_{avg} \+ (Expected\\\_Output\\\_Tokens \\times TPOT\_{avg})$$  
**3\. Экспоненциальное сглаживание метрик (EMA) для TTFT/TPOT:**  
Обновляется в core::state после каждого вызова.

$$New\\\_Metric \= (\\alpha \\times Current\\\_Execution\\\_Value) \+ ((1 \- \\alpha) \\times Old\\\_Metric)$$

### **VII. Пайплайн жизни задачи**

> 1. **Input:** Задача поступает в core::api.  
> 2. **Дедупликация:** Хэширование и поиск идентичного активного графа.  
> 3. **Ingress & Routing:** Задача без Template\_ID попадает в Узел Зеро для классификации.  
> 4. **Маршрутизация (Развилка):**  
   * **Fast Track (Статический):** Если ID найден, загружается YAML из .config/.  
   * **Deep Track (Динамический):** Если задача новая, запускается автономный пайплайн:  
     * *Фаза А (Context Architect):* Легкая модель получает задачу и .md файл с описанием инструментов. Возвращает JSON-список необходимых MCP-вызовов (без решения задачи). Ядро выполняет вызовы.  
     * *Фаза B (Pipeline Architect):* Умная модель получает задачу, логи от MCP и список доступных нейронок/агентов. Генерирует JSON, строго соответствующий схеме PipelineDefinition.  
     * *Фаза C (Validation):* Ядро проверяет схему сгенерированного графа. Если ошибка — заставляет модель переписать JSON.  
> 5. **Graph Compilation:** core::scheduler парсит YAML/JSON в DAG. Назначает каждому узлу $T\_{estimate}$.  
> 6. **Scheduling (WSJF):** Узлы без зависимостей переходят в Ready. Планировщик группирует узлы по требуемым моделям, вычисляет $Score$ с учетом $S\_{penalty}$.  
> 7. **Execution (IPC):** Пул с максимальным $Score$ уходит в работу. Ядро пишет промпт в shmem, поднимает флаг EXEC. Воркер выполняет инференс.  
> 8. **Streaming & Transition:** Воркер стримит результат. При наличии оператора ветвления (IF/ELSE) в ExecutionNode, Ядро валидирует результат и активирует соответствующую ветку графа.  
> 9. **Post-Processing/Testing:** Запускаются финальные агенты тестирования графа. Цикл замыкается.
